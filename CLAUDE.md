# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```
npm run dev       # start Vite dev server (defaults to :5173; if that port is busy Vite silently
                   # picks 5174/5175/... — always check the printed URL, don't assume 5173)
npm run build     # production build to dist/
npm run preview   # serve the production build locally
npm run lint      # oxlint (config: .oxlintrc.json)
```

There is no test suite/test script configured in this repo.

On Windows, a previous `npm run dev` that wasn't cleanly killed will keep holding its port and silently serve stale code to anything hitting `:5173`. If dev-server output looks wrong, check for orphaned `node.exe` processes (`Get-Process node`) before debugging the app itself.

`vite.config.js` sets `build.rollupOptions.output.manualChunks` as a **function**, not the plain `{ vendor: [...] }` object form — this Vite version builds on Rolldown, which throws (`TypeError: manualChunks is not a function`) if given the object form classic Rollup accepts. The function groups `react`/`react-dom`/`react-router-dom`/`@tanstack`/`axios`/`zustand`/`sonner` into one `vendor` chunk via a `node_modules/<pkg>/` regex, separate from app code — so a code change doesn't invalidate the browser cache for the (unchanged) vendor bundle. If you add a new major dependency that should also live in `vendor`, add it to the regex in `vite.config.js`, not by copying the old object syntax.

## What this is

A Tik Tak e-commerce **admin panel** UI built from a Figma design (Vite + React 19), wired to a real backend at `VITE_API_BASE_URL` (see `.env.example`; full endpoint reference in `docs/API.md`). All screens/text are Azerbaijani. Every list page fetches from the real API via **TanStack Query** (`useQuery`/`useMutation`, see below) — there is no offline/mock mode. Auth is real: the Login page calls `POST /auth/admin/login` via the `useAuthStore` zustand store (`src/store/useAuthStore.js`), and all `AdminLayout` routes are wrapped in a `RequireAuth` route guard (`src/routes/RequireAuth.jsx`) that redirects to `/login` when no access token is stored. Login itself reports a failed attempt (wrong password, empty fields) as a toast, same as every other page — no inline error text anywhere in the app. Its phone/password inputs are `lucide-react` icons absolutely positioned inside a `.inputWrap` — `Phone` fixed on the left of the phone field; the password field has both `Lock` fixed on the left AND a clickable `Eye`/`EyeOff` on the right that toggles `type="password"`↔`"text"`, so it carries all three of `.hasLeadingIcon` + `.hasTrailingIcon` + a `Lock` sibling at once. `.hasLeadingIcon`/`.hasTrailingIcon` on `.input` just add padding so the text doesn't sit under whichever icon(s) are present; reuse this wrapper pattern (not a new component) if another form ever needs an icon-in-input. Page components are lazy-loaded (`React.lazy` in `src/routes/AppRoutes.jsx`) so each route ships as its own JS chunk.

## Architecture

**Path alias**: `@` → `src/`, configured in **both** `vite.config.js` (`resolve.alias`) and `jsconfig.json` (`paths`) — keep them in sync if changed. All cross-folder imports use `@/...`; only same-folder sibling imports (e.g. a component importing its own `.module.css`) stay relative.

**Entry point**: `src/app/main.jsx` (referenced directly by path in `index.html`, not the `@` alias, since it's the one file Vite needs before any bundling/aliasing exists) renders `<ErrorBoundary><App/></ErrorBoundary>` — the boundary sits here, outside everything, so it also covers `App.jsx` itself and its providers, not just the routed pages. No `<StrictMode>` — removed deliberately, not an oversight. `src/app/App.jsx` mounts `<QueryClientProvider>` → `<BrowserRouter>` around `<AppRoutes />` + a sibling `<Toaster/>` (sonner).

**Routing**: the actual route tree lives in `src/routes/AppRoutes.jsx`, kept separate so `App.jsx` stays a thin shell. react-router-dom, Azerbaijani paths — `/login`, `/sifarisler`, `/kampaniyalar`, `/kateqoriyalar`, `/mehsullar`, `/istifadeciler`. `/login` is wrapped in `RedirectIfAuth` (bounces an already-logged-in admin to `/sifarisler`). All other routes are nested under a `RequireAuth` layout route (redirects to `/login` when unauthenticated), which itself nests them under `AdminLayout` via a layout route + `<Outlet>`. `/` redirects to `/sifarisler`; unmatched paths (`*`) render `src/pages/NotFound/NotFound.jsx` (a standalone full-page 404, not wrapped in `AdminLayout`, same pattern as `Login`) instead of redirecting — this is intentional so a broken/typo'd URL is visibly a 404 rather than silently bouncing to Orders. Auth state lives in a zustand store, not a React context/provider, so nothing wraps the router for auth. Every page component in `AppRoutes.jsx` is `lazy(() => import(...))`; the whole `<Routes>` tree is wrapped in one `<Suspense fallback={<Loading fullScreen/>}>` — reuses the same shared `Loading` spinner pages use for data fetches, so a route-chunk load and a slow API call look the same to the user; `fullScreen` centers it in the viewport since at that point it's the only thing rendered on the page, vs. the default inline mode pages use next to their own content. Don't add page imports as static `import Foo from '...'` — follow the `lazy(...)` pattern or the chunk-splitting benefit is lost for that route.

**Error handling**: three distinct, deliberately separate mechanisms — don't conflate them. (1) A bad **URL** (`*` route) renders `NotFound`. (2) A **JS exception thrown while rendering** anywhere in the tree is caught by `src/shared/ErrorBoundary/ErrorBoundary.jsx` (a class component using `getDerivedStateFromError`/`componentDidCatch`, the only place in the codebase that needs to be a class — error boundaries can't be hooks) wrapping `<App/>` in `src/app/main.jsx` (see **Entry point** above, not inside `App.jsx` — it needs to sit above the providers too). The fallback UI (icon + message + a plain `window.location.href = '/sifarisler'` button, not `useNavigate` — hooks aren't available in a class component's `render()`) lives directly inside `ErrorBoundary.jsx`/`ErrorBoundary.module.css`, there's no separate error page/route for it. (3) **API/network failures** are *not* funneled through the error boundary — `src/services/axiosInstance.js`'s `handleError` just normalizes them to a plain `Error` and rejects the promise; every query/mutation error is toasted **globally and automatically** by `src/lib/queryClient.js`'s `QueryCache`/`MutationCache` `onError` (`toast.error(err.message)`) — pages never write their own error-toast code, a bare `try { await mutateAsync(...) } catch {}` in a page is just there to stop `setFormOpen(false)`/`setDeleteTarget(null)` from running on failure, not to show anything. Success toasts are the one thing pages DO call explicitly, in each mutation's own `onSuccess`, phrased in the Azerbaijani completed/passive tense ("Kateqoriya yaradıldı", "Məhsul silindi", "Sifariş statusu yeniləndi") — never a progress tense ("yaradılır", "silinir"). `createMutation.isPending`/`updateMutation.isPending` (OR'd together) drives the submit button's `disabled` + "Göndərilir..." label — there's no separate hand-rolled `submitting` state anymore.

**Folder convention**: every component/page is its own folder containing `Name.jsx` + `Name.module.css` side by side — no bare `.jsx` files outside that pattern, no CSS-in-JS, no Tailwind.

- `src/shared/` — generic reusable UI primitives: `Button`, `Modal`/`ConfirmModal` (`role="dialog"`/`role="alertdialog"` + `aria-modal`, focus the close/cancel button on open, `Escape` key calls `onClose`/`onCancel` — both wired via a `useEffect` keyed on `open`, not a focus-trap library), `Badge`, `StatCard`, `ActionMenu`, `Table`, `Loading` (96px 4-color ring spinner — pure CSS, one `border` per side in a different badge color (green/blue/amber/purple), not an icon or a spinner library — + "Yüklənir..." text, stacked/centered layout, not inline; `fullScreen` prop adds `min-height: 70vh` to center it in the viewport for the route-level `Suspense` fallback — default/no-prop mode is what pages use for their own `isLoading` query state, see **Routing**), `Thumbnail` (the `imageUrl` → `<img>` / fallback-emoji-over-`color` pattern from **CRUD modal pattern** below, `size="sm"` for table cells (default) or `size="lg"` for detail-modal headers), `ErrorBoundary` (the one class component in the app — see **Error handling**), plus `src/shared/hooks/` for cross-cutting hooks (`useTitle`, `useDebounce`, `usePagination`, `useCrudModal`) — flat `.js` files, no per-hook folder/CSS since hooks have no markup. There is no `ErrorText`/inline-error component and no hand-rolled data-fetching hook — data fetching is TanStack Query (`useQuery`/`useMutation`, see **Data fetching** below), all error/success feedback is a toast (see **Error handling**).
- `src/utils/` — `Pagination` lives here by project convention, not in `shared/`.
- `src/components/` — layout-only chrome (`Sidebar`, `Header`), used exclusively by `AdminLayout`.
- `src/layouts/` — the sidebar+header shell wrapping every non-login page. `AdminLayout.jsx` + `AdminLayout.module.css` sit flat directly in `src/layouts/` (no nested `AdminLayout/` subfolder) since it's the only layout in the app — add a per-layout subfolder only if a second layout is ever introduced.
- `src/pages/` — one folder per route. `src/pages/protected/<Name>/` (`Orders`, `Campaigns`, `Categories`, `Products`, `Users`) holds every page behind `RequireAuth`; put any new authenticated page there. `src/pages/Login/` sits at the top level, unwrapped — it's the one route wrapped in `RedirectIfAuth` instead of `RequireAuth`, so grouping it under a generic folder wasn't worth it for a single page.
- `src/services/` — the API layer. `axiosInstance.js` creates one shared `axios.create()` instance and holds the only two response handlers (`handleSuccess`/`handleError`, both a few lines); every other service file (`categoryService.js`, `productService.js`, `campaignService.js`, `orderService.js`, `userService.js`, `authService.js`, `uploadService.js`) just calls `api.get/post/put/delete` on it — no per-file error handling, no custom client class.
- `src/lib/adapters/` — one file per resource (`category.js`, `product.js`, `campaign.js`, `order.js`, `user.js`) with `mapXFromApi`/`mapXToApi` pairs that translate between the API's field names (snake_case, `title` vs `name`, `category_id` vs nested `category`) and the shape pages/JSX expect.
- `src/lib/auth/session.js` — localStorage read/write/clear for the access/refresh tokens and cached profile.
- `src/lib/constants/` — `productTypes.js` (`ProductMeasure` enum → Azerbaijani labels + badge color) and `orderStatus.js` (`OrderStatus` enum → Azerbaijani labels + badge color).
- `src/store/useAuthStore.js` — zustand store (`create`, no Provider needed), owns `login`/`logout`/`profile`/`isAuthenticated`. Also registers a module-level `window.addEventListener('storage', ...)` that re-syncs `isAuthenticated`/`profile` from `localStorage` whenever another tab logs in/out — the browser only fires `storage` on tabs *other than* the one that made the change, so this is what keeps a second open tab from staying "logged in" after you log out in the first one; `RequireAuth` picks up the resulting state change automatically since it already subscribes to `isAuthenticated`.
- `src/routes/` — `RequireAuth.jsx` and `RedirectIfAuth.jsx` route guards used in `App.jsx`, both read `useAuthStore` directly.
- `src/app/` — `main.jsx` (entry point, referenced by `index.html`) and `App.jsx` (providers + router shell) — the only two files in the whole codebase not under `src/pages|shared|components|...`, kept together since they're both "bootstrap the app" concerns, not features.
- `src/assets/images/` — SVG illustrations from the Figma design (`login-img.svg`, `delete-img.svg`), imported and rendered directly as `<img src={...}>`.

**API layer conventions** (see `docs/API.md` for the full endpoint reference): base URL comes from `VITE_API_BASE_URL` (`.env`, not committed — see `.env.example`). `src/services/axiosInstance.js` is the single place that knows about HTTP: a request interceptor attaches `Authorization: Bearer <token>`, and the response interceptor's `handleSuccess` unwraps the standard `{message, data, result}` envelope (falls back to the raw body when there's no `data` key, e.g. `GET /orders/admin/stats`) so every service function already resolves to the usable payload — callers never touch `response.data`. On a `401`, `handleError` transparently refreshes the access token via `POST /auth/refresh` (`refreshAccessToken()`, concurrent 401s share one in-flight refresh call via `refreshPromise` so a burst of requests doesn't fire the refresh endpoint multiple times) and retries the original request exactly once (`original._retry` guards against a retry loop); if the refresh itself fails, or there's no refresh token stored, it calls `clearSession()` + `useAuthStore.getState().logout()`. `POST /auth/admin/login` is the one request that opts out of this via `{ skipAuthRetry: true }` in `authService.js` — a wrong-password 401 on login must not trigger a refresh attempt or a logout. The backend's own error messages come back in **English** (e.g. `"Password is wrong!"`, `"Auth access error"`) — `handleError` never surfaces them directly; `getErrorMessage()` maps the HTTP status code to a fixed Azerbaijani string instead (`STATUS_MESSAGES`, plus a special case for `401` that reads "Telefon və ya parol yanlışdır" on the login request vs "Sessiya bitib, yenidən daxil olun" everywhere else), so every toast in the app — success or error — is guaranteed to be in Azerbaijani regardless of what the backend actually said. When adding a new endpoint, don't reach for `error.response.data.message` — extend `STATUS_MESSAGES` if a new status code needs its own wording. Every other failure just rejects the promise — no redirect, no error-boundary involvement (see **Error handling** above for why those two are kept separate from plain API errors). Category/Product/Campaign **create** endpoints use a **singular** URL (`/admin/category`) while **update/delete/list** use the **plural** form (`/admin/categories/:id`) — this is a real backend quirk, not a typo; each `src/services/*Service.js` file already encodes it correctly, don't "fix" it to be consistent.

**Data fetching** (TanStack Query, `src/lib/queryClient.js`): `defaultOptions.queries.staleTime` is set to 15s (not TanStack's default of 0) — navigating back to a page you were just on within 15s reuses the cached data instead of refetching; a create/update/delete's `invalidateQueries` still forces an immediate refetch regardless of `staleTime`, so mutations always show fresh data. Every list page reads via `useQuery({ queryKey: [...], queryFn: ... })` and writes via `useMutation`, invalidating the relevant `queryKey` in `onSuccess` instead of hand-patching local array state. Query keys are the resource name in English, plural, matching the service module (`['categories']`, `['products']`, `['campaigns']`, `['orders']`, `['orderStats']`, `['users']`) — reuse the exact same key when two pages need the same resource: `Products.jsx` queries `['categories']` for its category dropdown with the *same key* `Categories.jsx` uses for its own list, so TanStack Query dedupes/shares that cache entry between the two pages instead of fetching it twice. **The `queryFn` for a shared key must produce the identical shape everywhere it's used** — TanStack Query caches by key alone, not by `queryFn`, so if two components register different mappings under the same key, whichever mounts first "wins" and the other silently renders the wrong shape until the cache goes stale (this was a real bug: `Products.jsx` used to fetch raw `listCategories()` under `['categories']` while `Categories.jsx` mapped it through `mapCategoryFromApi` under the same key — fixed by making both use the mapped version). `src/components/Sidebar/Sidebar.jsx`'s `PREFETCH` map calls `queryClient.prefetchQuery` on `onMouseEnter` for each nav link, using these exact same `queryKey`/`queryFn` pairs, for the same reason — copy them from the page, don't invent a different shape "close enough" for prefetching. `Orders.jsx` similarly runs two independent `useQuery` calls (`['orders']`, `['orderStats']`) rather than one combined `Promise.all` fetch — simpler than trying to merge two unrelated pieces of state into one query result. Mutation success/error are NOT handled per-page (see **Error handling**); the only page-level `try/catch` around `mutateAsync` exists to gate closing the modal/dialog on success. `Orders.jsx`'s detail modal keeps a `selectedId` (not a snapshot object) and derives `selected = orders.find(o => o.id === selectedId)` — after a status-change mutation invalidates and refetches `['orders']`, `selected` automatically reflects the fresh data (including `user`, which the status-update response itself omits but the list response always includes) with no manual merging.

**Fonts**: `src/index.css` imports only the `latin` + `latin-ext` subsets of `@fontsource/roboto` (`latin-300.css`, `latin-ext-300.css`, `latin-400.css`, ... for weights 300/400/500/700) — not the bare `300.css`/`400.css`/etc., which pull in *every* subset the font ships (cyrillic, cyrillic-ext, greek, greek-ext, vietnamese, symbols, math) and roughly quadruple the font payload for a UI that's Azerbaijani-only. `latin-ext` specifically covers `ə/ğ/ı/ö/ş/ü/ç`. If a new weight is ever needed, import both its `latin-`/`latin-ext-` files, not the unscoped one.

**Design tokens**: all colors, radii, shadows, and shared layout dimensions are CSS custom properties defined once in `src/index.css` (`:root`), e.g. `--color-green`, `--radius-sm`, `--shell-max-width`, `--table-row-height`, `--page-title-height`. Page modules reference these vars instead of hardcoding values — reuse them rather than introducing new ad-hoc constants. `index.css` also globally resets `margin: 0` on `h1–h6, p, dl, dd`; don't rely on default browser margins on these elements inside flex/grid layouts (a real layout bug here — mismatched heading margins — previously caused table containers to misalign between pages).

**`shared/Table`**: a generic, data-driven table shell. Pages pass a `columns` array of `{ key, label, width }` and render `<tr>` children directly; `Table` renders the `<thead>` from `columns`, wraps everything in a horizontal-scroll container, and uses `table-layout: fixed`. **Every column except exactly one must have an explicit `width`** (px or `%`) — leaving two or more columns unconstrained makes column widths jump around whenever the row content/filter changes (this exact bug happened before and was fixed by auditing every `columns` array).

**Consistent page chrome via CSS vars**: `--shell-content-height` keeps every list page's white container the same height regardless of row count; `--table-row-height` keeps table rows a fixed height; `--page-title-height` keeps a page's title row the same height whether or not it includes a "Yeni X" add button. When adding a new list page, reuse these vars rather than hand-tuning heights, or the pages will visually drift apart again.

**`shared/ActionMenu`**: a 3-dot trigger whose dropdown is rendered via `createPortal(..., document.body)` — required because table cells sit inside an `overflow-x: auto` scroll container that would otherwise clip an absolutely-positioned dropdown. Used only on **Campaigns / Categories / Products**; **Orders** and **Users** intentionally use a plain inline "Göstər" button + detail `Modal` instead (a deliberate product inconsistency, not an oversight — don't "fix" it by unifying them).

**CRUD modal pattern** (Campaigns, Categories, Products all repeat this shape — copy it for any new similar page): all the state behind it — `formOpen` / `editing` / `form` / `deleteTarget` / `viewTarget`, plus `openCreate`/`openEdit` — comes from `useCrudModal(emptyForm, toForm)` (`src/shared/hooks/useCrudModal.js`), not hand-rolled per page. `toForm(item)` maps a list item to the form shape (mirrors the page's own `emptyForm` shape); `openCreate(overrides?)` takes an optional partial-form object for cases like Products, where the create form needs a default `category_id` computed from data the hook itself doesn't know about (`openCreate({ category_id: categoryOptions[0]?.id ?? '' })`). On top of that:
- one `Modal` for create/edit, driven by `formOpen` / `editing` / `form`,
- one `ConfirmModal` for delete, driven by `deleteTarget`,
- one `wide` `Modal` for a read-only detail view, driven by `viewTarget`, opened from `ActionMenu`'s "Bax".

Pagination (`page`/`pageSize`/`paged`) on every list page — including Orders and Users, which don't use `useCrudModal` — comes from `usePagination(filteredItems, initialPageSize?)` (`src/shared/hooks/usePagination.js`): slices `filteredItems` client-side and resets `page` to `1` whenever `setPageSize` is called (Orders is the only page with a page-size selector, via `Pagination`'s `onPageSizeChange`).

The "Şəkil ünvanı" field in these forms is a plain text input bound to `form.imageUrl`, mapped to/from the API's `img_url` by the relevant `src/lib/adapters/*.js` file. Thumbnails are rendered by the shared `Thumbnail` component (`imageUrl`/`image`/`color` props): `<img>` if `imageUrl`/`img_url` is set, otherwise a fixed decorative emoji+color pair defined per-resource in the adapter (the API itself carries no emoji/color field — there are no real image files behind most items yet). `src/services/uploadService.js` (`POST /upload`, multipart) exists for future use but is not wired into any form UI yet — the image field stays a manual URL text input for now.

**Search flow**: the raw search string lives in `AdminLayout` (`useState`, passed to `Header`'s controlled input so typing feels instant), but the value handed down to pages via `<Outlet context={{ search }}>` is `useDebounce(search, 500)` — so every page's `useMemo` filter (reading `search` via `useOutletContext()`) only re-runs 500ms after the user stops typing. Don't debounce inside individual pages; the debounce point is centralized in `AdminLayout` so all pages get it for free.

**`useTitle(title)`** (`src/shared/hooks/useTitle.js`): sets `document.title` to `"{title} · Tik Tak Admin"` for the life of the component and restores the previous title on unmount. Every page component calls it once, near the top, with its own Azerbaijani title (e.g. `useTitle('Sifarişlər')`) — do the same for any new page.
