# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```
npm run dev        # start Vite dev server (defaults to :5173; if that port is busy Vite silently
                    # picks 5174/5175/... — always check the printed URL, don't assume 5173)
npm run build      # production build to dist/ (runs Vite/esbuild's own TS transpile — does NOT
                    # type-check; a type error will NOT fail this command, see typecheck below)
npm run preview    # serve the production build locally
npm run lint       # oxlint (config: .oxlintrc.json) — also lints TS/TSX, see TypeScript section
npm run typecheck  # tsc --noEmit — the ONLY command that actually checks types; run this before
                    # claiming a change is done, build succeeding is not proof the types are right
```

There is no test suite/test script configured in this repo.

On Windows, a previous `npm run dev` that wasn't cleanly killed will keep holding its port and silently serve stale code to anything hitting `:5173`. If dev-server output looks wrong, check for orphaned `node.exe` processes (`Get-Process node`) before debugging the app itself.

`vite.config.js` sets `build.rollupOptions.output.manualChunks` as a **function**, not the plain `{ vendor: [...] }` object form — this Vite version builds on Rolldown, which throws (`TypeError: manualChunks is not a function`) if given the object form classic Rollup accepts. The function groups `react`/`react-dom`/`react-router-dom`/`@tanstack`/`axios`/`zustand`/`sonner` into one `vendor` chunk via a `node_modules/<pkg>/` regex, separate from app code — so a code change doesn't invalidate the browser cache for the (unchanged) vendor bundle. If you add a new major dependency that should also live in `vendor`, add it to the regex in `vite.config.js`, not by copying the old object syntax.

`vite.config.js`'s `plugins` array also includes `tailwindcss()` from `@tailwindcss/vite` (alongside `react()`), and `src/index.css`'s first line is `@import "tailwindcss";` — both are required together, removing either one silently breaks every Tailwind utility class in the app. See **Styling: CSS Modules + Tailwind** below for what belongs in which system.

## What this is

A Tik Tak e-commerce **admin panel** UI built from a Figma design (Vite + React 19 + **TypeScript**), wired to a real backend at `VITE_API_BASE_URL` (see `.env.example`; full endpoint reference in `docs/API.md`). All screens/text are Azerbaijani. Every list page fetches from the real API via **TanStack Query** (`useQuery`/`useMutation`, see below) — there is no offline/mock mode. Auth is real: the Login page calls `POST /auth/admin/login` via the `useAuthStore` zustand store (`src/store/useAuthStore.ts`), and all `AdminLayout` routes are wrapped in a `RequireAuth` route guard (`src/routes/RequireAuth.tsx`) that redirects to `/login` when no access token is stored. Login itself reports a failed attempt (wrong password, empty fields) as a toast, same as every other page — no inline error text anywhere in the app. Its phone/password inputs are `lucide-react` icons absolutely positioned inside a `.inputWrap` — `Phone` fixed on the left of the phone field; the password field has both `Lock` fixed on the left AND a clickable `Eye`/`EyeOff` on the right that toggles `type="password"`↔`"text"`, so it carries all three of `.hasLeadingIcon` + `.hasTrailingIcon` + a `Lock` sibling at once. `.hasLeadingIcon`/`.hasTrailingIcon` on `.input` just add padding so the text doesn't sit under whichever icon(s) are present; reuse this wrapper pattern (not a new component) if another form ever needs an icon-in-input. Page components are lazy-loaded (`React.lazy` in `src/routes/AppRoutes.tsx`) so each route ships as its own JS chunk.

The whole `src/` tree is TypeScript (`.ts`/`.tsx`, zero `.js`/`.jsx` remaining) — see the dedicated **TypeScript** section below for the type policy, `src/types/` layout, and the handful of genuinely advanced spots (module augmentation, the `api` cast) before touching typed code.

## Architecture

**Path alias**: `@` → `src/`, configured in **both** `vite.config.js` (`resolve.alias`) and `tsconfig.json` (`paths`) — keep them in sync if changed. All cross-folder imports use `@/...`; only same-folder sibling imports (e.g. a component importing its own `.module.css`) stay relative. There is no `jsconfig.json` — it was deleted when `tsconfig.json` took over path resolution; don't re-add it.

**Entry point**: `src/app/main.tsx` (referenced directly by path in `index.html`, not the `@` alias, since it's the one file Vite needs before any bundling/aliasing exists) renders `<ErrorBoundary><App/></ErrorBoundary>` — the boundary sits here, outside everything, so it also covers `App.tsx` itself and its providers, not just the routed pages. No `<StrictMode>` — removed deliberately, not an oversight. `src/app/App.tsx` mounts `<QueryClientProvider>` → `<BrowserRouter>` around `<AppRoutes />` + a sibling `<Toaster/>` (sonner).

**Routing**: the actual route tree lives in `src/routes/AppRoutes.tsx`, kept separate so `App.tsx` stays a thin shell. react-router-dom, Azerbaijani paths — `/login`, `/sifarisler`, `/kampaniyalar`, `/kateqoriyalar`, `/mehsullar`, `/istifadeciler`. `/login` is wrapped in `RedirectIfAuth` (bounces an already-logged-in admin to `/sifarisler`). All other routes are nested under a `RequireAuth` layout route (redirects to `/login` when unauthenticated), which itself nests them under `AdminLayout` via a layout route + `<Outlet>`. `/` redirects to `/sifarisler`; unmatched paths (`*`) render `src/pages/NotFound/NotFound.tsx` (a standalone full-page 404, not wrapped in `AdminLayout`, same pattern as `Login`) instead of redirecting — this is intentional so a broken/typo'd URL is visibly a 404 rather than silently bouncing to Orders. Auth state lives in a zustand store, not a React context/provider, so nothing wraps the router for auth. Every page component in `AppRoutes.tsx` is `lazy(() => import(...))`; the whole `<Routes>` tree is wrapped in one `<Suspense fallback={<Loading fullScreen/>}>` — reuses the same shared `Loading` spinner pages use for data fetches, so a route-chunk load and a slow API call look the same to the user; `fullScreen` centers it in the viewport since at that point it's the only thing rendered on the page, vs. the default inline mode pages use next to their own content. Don't add page imports as static `import Foo from '...'` — follow the `lazy(...)` pattern or the chunk-splitting benefit is lost for that route.

**Error handling**: three distinct, deliberately separate mechanisms — don't conflate them. (1) A bad **URL** (`*` route) renders `NotFound`. (2) A **JS exception thrown while rendering** anywhere in the tree is caught by `src/shared/components/ErrorBoundary/ErrorBoundary.tsx` (a class component using `getDerivedStateFromError`/`componentDidCatch`, the only place in the codebase that needs to be a class — error boundaries can't be hooks; its `override` keywords on `state`/`componentDidCatch`/`render` are required by `noImplicitOverride`, see **TypeScript**) wrapping `<App/>` in `src/app/main.tsx` (see **Entry point** above, not inside `App.tsx` — it needs to sit above the providers too). The fallback UI (icon + message + a plain `window.location.href = '/sifarisler'` button, not `useNavigate` — hooks aren't available in a class component's `render()`) lives directly inside `ErrorBoundary.tsx`/`ErrorBoundary.module.css`, there's no separate error page/route for it. (3) **API/network failures** are *not* funneled through the error boundary — `src/services/axiosInstance.ts`'s `handleError` just normalizes them to a plain `Error` and rejects the promise; every query/mutation error is toasted **globally and automatically** by `src/lib/queryClient.ts`'s `QueryCache`/`MutationCache` `onError` (`toast.error(err.message)`) — pages never write their own error-toast code, a bare `try { await mutateAsync(...) } catch {}` in a page is just there to stop `setFormOpen(false)`/`setDeleteTarget(null)` from running on failure, not to show anything. Success toasts are the one thing pages DO call explicitly, in each mutation's own `onSuccess`, phrased in the Azerbaijani completed/passive tense ("Kateqoriya yaradıldı", "Məhsul silindi", "Sifariş statusu yeniləndi") — never a progress tense ("yaradılır", "silinir"). `createMutation.isPending`/`updateMutation.isPending` (OR'd together) drives the submit button's `disabled` + "Göndərilir..." label — there's no separate hand-rolled `submitting` state anymore.

**Folder convention**: every component/page is its own folder containing `Name.tsx` + `Name.module.css` side by side — no bare `.tsx` files outside that pattern, no CSS-in-JS. Plain data/logic files (services, adapters, hooks, constants) are `.ts`; anything that returns JSX is `.tsx`.

- `src/shared/components/` — generic reusable UI primitives: `Button`, `Modal`/`ConfirmModal` (`role="dialog"`/`role="alertdialog"` + `aria-modal`, focus the close/cancel button on open, `Escape` key calls `onClose`/`onCancel` — both wired via a `useEffect` keyed on `open`, not a focus-trap library; `ConfirmModal` takes an optional `showIcon?: boolean`, default `true` — the delete confirmations on Campaigns/Categories/Products don't pass it, so they keep the `delete-img.svg` icon, while `Sidebar`'s logout confirmation passes `showIcon={false}` since logging out isn't a destructive/delete action), `Badge`, `StatCard`, `ActionMenu`, `Table`, `Loading` (96px 4-color ring spinner — pure CSS, one `border` per side in a different badge color (green/blue/amber/purple), not an icon or a spinner library — + "Yüklənir..." text, stacked/centered layout, not inline; `fullScreen` prop adds `min-height: 70vh` to center it in the viewport for the route-level `Suspense` fallback — default/no-prop mode is what pages use for their own `isLoading` query state, see **Routing**), `Thumbnail` (the `imageUrl` → `<img>` / fallback-emoji-over-`color` pattern from **CRUD modal pattern** below, `size="sm"` for table cells (default) or `size="lg"` for detail-modal headers), `ErrorBoundary` (the one class component in the app — see **Error handling**). `src/shared/hooks/` sits as a sibling of `shared/components/` (not inside it) for cross-cutting hooks (`useTitle`, `useDebounce`, `usePagination`, `useCrudModal`) — flat `.ts` files, no per-hook folder/CSS since hooks have no markup; `usePagination`/`useDebounce`/`useCrudModal` are generic (`usePagination<T>`, etc., see **TypeScript**). There is no `ErrorText`/inline-error component and no hand-rolled data-fetching hook — data fetching is TanStack Query (`useQuery`/`useMutation`, see **Data fetching** below), all error/success feedback is a toast (see **Error handling**).
- `src/types/` — the TypeScript type definitions, one file per resource (`category.ts`, `product.ts`, `campaign.ts`, `order.ts`, `user.ts`, plus `auth.ts`, `api.ts`, `common.ts`) and a thin `index.ts` barrel. See the **TypeScript** section below for what belongs here vs. what stays colocated with the code it describes.
- `src/utils/` — `Pagination` lives here by project convention, not in `shared/`.
- `src/components/` — layout-only chrome (`Sidebar`, `Header`), used exclusively by `AdminLayout`. `Sidebar`'s "Çıxış" button doesn't call `logout()` directly — it opens a `ConfirmModal` (`showIcon={false}`, see above) first; only its `onConfirm` calls `logout()` + `navigate('/login')`, and `onCancel` just closes the modal. Don't "simplify" this back to a bare `onClick={logout}` — the confirm step is intentional, not leftover scaffolding.
- `src/layouts/` — the sidebar+header shell wrapping every non-login page. `AdminLayout.tsx` + `AdminLayout.module.css` sit flat directly in `src/layouts/` (no nested `AdminLayout/` subfolder) since it's the only layout in the app — add a per-layout subfolder only if a second layout is ever introduced.
- `src/pages/` — one folder per route. `src/pages/protected/<Name>/` (`Orders`, `Campaigns`, `Categories`, `Products`, `Users`) holds every page behind `RequireAuth`; put any new authenticated page there. `src/pages/Login/` sits at the top level, unwrapped — it's the one route wrapped in `RedirectIfAuth` instead of `RequireAuth`, so grouping it under a generic folder wasn't worth it for a single page.
- `src/services/` — the API layer. `axiosInstance.ts` creates one shared `axios.create()` instance and holds the only two response handlers (`handleSuccess`/`handleError`, both a few lines); every other service file (`categoryService.ts`, `productService.ts`, `campaignService.ts`, `orderService.ts`, `userService.ts`, `authService.ts`, `uploadService.ts`) just calls `api.get<T>/post<T>/put<T>/delete<T>` on it — no per-file error handling, no custom client class. See **TypeScript** for what `T` is doing there.
- `src/lib/adapters/` — one file per resource (`category.ts`, `product.ts`, `campaign.ts`, `order.ts`, `user.ts`) with `mapXFromApi`/`mapXToApi` pairs that translate between the API's field names (snake_case, `title` vs `name`, `category_id` vs nested `category`) and the shape pages/JSX expect. Each pair is typed `(x: XApi) => X` / `(form: XForm) => XPayload`, all four types imported from `src/types/`.
- `src/lib/auth/session.ts` — localStorage read/write/clear for the access/refresh tokens and cached profile.
- `src/lib/constants/` — `productTypes.ts` (`ProductType` union → Azerbaijani labels + badge color) and `orderStatus.ts` (`OrderStatus` union → Azerbaijani labels + badge color). These are the **single source of truth** for the two union types — `src/types/product.ts`/`order.ts` re-export them type-only rather than redefining, see **TypeScript**.
- `src/store/useAuthStore.ts` — zustand store (`create<AuthState>()`, no Provider needed), owns `login`/`logout`/`profile`/`isAuthenticated`. Also registers a module-level `window.addEventListener('storage', ...)` that re-syncs `isAuthenticated`/`profile` from `localStorage` whenever another tab logs in/out — the browser only fires `storage` on tabs *other than* the one that made the change, so this is what keeps a second open tab from staying "logged in" after you log out in the first one; `RequireAuth` picks up the resulting state change automatically since it already subscribes to `isAuthenticated`.
- `src/routes/` — `RequireAuth.tsx` and `RedirectIfAuth.tsx` route guards used in `App.tsx`, both read `useAuthStore` directly.
- `src/app/` — `main.tsx` (entry point, referenced by `index.html`) and `App.tsx` (providers + router shell) — the only two files in the whole codebase not under `src/pages|shared|components|...`, kept together since they're both "bootstrap the app" concerns, not features.
- `src/assets/images/` — SVG illustrations from the Figma design (`login-img.svg`, `delete-img.svg`), imported and rendered directly as `<img src={...}>`.

**API layer conventions** (see `docs/API.md` for the full endpoint reference): base URL comes from `VITE_API_BASE_URL` (`.env`, not committed — see `.env.example`). `src/services/axiosInstance.ts` is the single place that knows about HTTP: a request interceptor attaches `Authorization: Bearer <token>`, and the response interceptor's `handleSuccess` unwraps the standard `{message, data, result}` envelope (falls back to the raw body when there's no `data` key, e.g. `GET /orders/admin/stats`) so every service function already resolves to the usable payload — callers never touch `response.data`. On a `401`, `handleError` transparently refreshes the access token via `POST /auth/refresh` (`refreshAccessToken()`, concurrent 401s share one in-flight refresh call via `refreshPromise` so a burst of requests doesn't fire the refresh endpoint multiple times) and retries the original request exactly once (`original._retry` guards against a retry loop); if the refresh itself fails, or there's no refresh token stored, it calls `clearSession()` + `useAuthStore.getState().logout()`. `POST /auth/admin/login` is the one request that opts out of this via `{ skipAuthRetry: true }` in `authService.ts` — a wrong-password 401 on login must not trigger a refresh attempt or a logout (`skipAuthRetry`/`_retry` are not real Axios fields — they're added to `AxiosRequestConfig` via a `declare module 'axios'` augmentation in `src/types/api.ts`, see **TypeScript**). The backend's own error messages come back in **English** (e.g. `"Password is wrong!"`, `"Auth access error"`) — `handleError` never surfaces them directly; `getErrorMessage()` maps the HTTP status code to a fixed Azerbaijani string instead (`STATUS_MESSAGES`, plus a special case for `401` that reads "Telefon və ya parol yanlışdır" on the login request vs "Sessiya bitib, yenidən daxil olun" everywhere else, and a special case for a `400`/`422` on a product-delete specifically — see the FK-constraint comment in `getErrorMessage`), so every toast in the app — success or error — is guaranteed to be in Azerbaijani regardless of what the backend actually said. When adding a new endpoint, don't reach for `error.response.data.message` — extend `STATUS_MESSAGES` if a new status code needs its own wording. Every other failure just rejects the promise — no redirect, no error-boundary involvement (see **Error handling** above for why those two are kept separate from plain API errors). Category/Product/Campaign **create** endpoints use a **singular** URL (`/admin/category`) while **update/delete/list** use the **plural** form (`/admin/categories/:id`) — this is a real backend quirk, not a typo; each `src/services/*Service.ts` file already encodes it correctly, don't "fix" it to be consistent.

**Data fetching** (TanStack Query, `src/lib/queryClient.ts`): `defaultOptions.queries.staleTime` is set to 15s (not TanStack's default of 0) — navigating back to a page you were just on within 15s reuses the cached data instead of refetching; a create/update/delete's `invalidateQueries` still forces an immediate refetch regardless of `staleTime`, so mutations always show fresh data. Every list page reads via `useQuery({ queryKey: [...], queryFn: ... })` and writes via `useMutation`, invalidating the relevant `queryKey` in `onSuccess` instead of hand-patching local array state. Query keys are the resource name in English, plural, matching the service module (`['categories']`, `['products']`, `['campaigns']`, `['orders']`, `['orderStats']`, `['users']`) — reuse the exact same key when two pages need the same resource: `Products.tsx` queries `['categories']` for its category dropdown with the *same key* `Categories.tsx` uses for its own list, so TanStack Query dedupes/shares that cache entry between the two pages instead of fetching it twice. **The `queryFn` for a shared key must produce the identical shape everywhere it's used** — TanStack Query caches by key alone, not by `queryFn`, so if two components register different mappings under the same key, whichever mounts first "wins" and the other silently renders the wrong shape until the cache goes stale (this was a real bug: `Products.tsx` used to fetch raw `listCategories()` under `['categories']` while `Categories.tsx` mapped it through `mapCategoryFromApi` under the same key — fixed by making both use the mapped version; TypeScript does NOT catch this class of bug on its own, since both queryFns still type-check individually — the discipline of reusing the exact same queryFn is still on you). `src/components/Sidebar/Sidebar.tsx`'s `PREFETCH` map calls `queryClient.prefetchQuery` on `onMouseEnter` for each nav link, using these exact same `queryKey`/`queryFn` pairs, for the same reason — copy them from the page, don't invent a different shape "close enough" for prefetching. `Orders.tsx` similarly runs two independent `useQuery` calls (`['orders']`, `['orderStats']`) rather than one combined `Promise.all` fetch — simpler than trying to merge two unrelated pieces of state into one query result; `orderStats`'s query is typed `Partial<OrderStats>` at the fetch boundary (see **TypeScript**) because the backend can omit counters. Mutation success/error are NOT handled per-page (see **Error handling**); the only page-level `try/catch` around `mutateAsync` exists to gate closing the modal/dialog on success. `Orders.tsx`'s detail modal keeps a `selectedId` (not a snapshot object) and derives `selected = orders.find(o => o.id === selectedId)` — after a status-change mutation invalidates and refetches `['orders']`, `selected` automatically reflects the fresh data (including `user`, which the status-update response itself omits but the list response always includes) with no manual merging. The status-update mutation is also **optimistic**: `onMutate` patches the cached `['orders']` entry immediately (so the status `<select>`/badge update with no visible lag) and rolls back via the `previousOrders` returned from `onMutate` if the request fails — see the comment on `statusMutation` in `Orders.tsx` for why this exists (a real UX bug: without it, the dropdown visibly "snapped back" for ~2s while the network round-trip completed).

**`Orders.tsx`'s status `<select>` color**: `ORDER_STATUS_BADGE_COLOR[status]` (`lib/constants/orderStatus.ts`) returns a `BadgeColor` key like `'amber'`/`'purple'` — these are **not** valid CSS color keywords (`'amber'` isn't a real CSS color at all), so they can't go straight into `style={{ color }}`. `Orders.tsx` defines a local `STATUS_TEXT_COLOR: Record<BadgeColor, string>` map that resolves each `BadgeColor` key to the same CSS var `Badge.module.css` itself uses (e.g. `amber` → `var(--color-amber-text)`) — reuse this map (or extend it) rather than passing a `BadgeColor` string directly as a CSS color anywhere else in the app. The closed `<select>` gets `style={{ color: STATUS_TEXT_COLOR[...] }}`, but each `<option>` inside it needs its own explicit `style={{ color: 'var(--color-text)' }}` — otherwise `color` is inherited from the parent `<select>`, so every option in the open dropdown list would render in whatever color the *currently selected* status happens to be, not just the closed control. `Orders.module.css`'s `.statusSelect` sets `border: 1px solid currentColor` (not a fixed var) plus `outline: none` — `currentColor` keeps the border matched to the dynamic status color in both the focused and unfocused state, so clicking away doesn't visibly change the border; don't reintroduce a separate `:focus` rule for `border-color`, it would just fight this.

**Fonts**: `src/index.css` imports only the `latin` + `latin-ext` subsets of `@fontsource/roboto` (`latin-300.css`, `latin-ext-300.css`, `latin-400.css`, ... for weights 300/400/500/700) — not the bare `300.css`/`400.css`/etc., which pull in *every* subset the font ships (cyrillic, cyrillic-ext, greek, greek-ext, vietnamese, symbols, math) and roughly quadruple the font payload for a UI that's Azerbaijani-only. `latin-ext` specifically covers `ə/ğ/ı/ö/ş/ü/ç`. If a new weight is ever needed, import both its `latin-`/`latin-ext-` files, not the unscoped one.

**Design tokens**: all colors, radii, shadows, and shared layout dimensions are CSS custom properties defined once in `src/index.css` (`:root`), e.g. `--color-green`, `--radius-sm`, `--shell-max-width`, `--table-row-height`, `--page-title-height`. Page modules reference these vars instead of hardcoding values — reuse them rather than introducing new ad-hoc constants. `index.css` also globally resets `margin: 0` on `h1–h6, p, dl, dd`; don't rely on default browser margins on these elements inside flex/grid layouts (a real layout bug here — mismatched heading margins — previously caused table containers to misalign between pages).

**Styling: CSS Modules + Tailwind**: the app uses **both**, each for a different job — don't blur the line between them. Tailwind utility classes (`flex`, `flex-col`, `items-center`, `justify-between`, `flex-wrap`, `gap-1`...`gap-6`/`gap-x-*`/`gap-y-*`, `text-center`/`text-left`/`text-right`, `cursor-pointer`, `resize-none`, `whitespace-nowrap`, `overflow-hidden`, and `truncate` for the `overflow:hidden;text-overflow:ellipsis;white-space:nowrap` triple) cover short, repetitive **layout** declarations and are appended directly in `className` template strings alongside the CSS Module class, e.g. `className={`flex items-center gap-3 ${styles.detailTop}`}`. Everything else — colors, `border-radius`, `box-shadow`, font sizes, one-off positioning, and any CSS tied to an **element-descendant selector** rather than a specific `className` (e.g. `Table.module.css`'s `.table th`/`.table td`, or `AdminLayout.module.css`'s `.main > *`) — stays in `.module.css` files using the design-token vars above; these were deliberately left alone since there's no single React element to attach a Tailwind class to. When picking a `gap-*`/`gap-x-*`/`gap-y-*` value, it must be an **exact** match to the existing CSS px value (Tailwind's default scale is `0.25rem` increments: `gap-1`=4px, `gap-1.5`=6px, `gap-2`=8px, `gap-2.5`=10px, `gap-3`=12px, `gap-4`=16px, `gap-5`=20px, `gap-6`=24px) — don't approximate to the nearest step, that silently changes spacing.

**`shared/components/Table`**: a generic, data-driven table shell. Pages pass a `columns: Column[]` array of `{ key, label, width }` (the `Column` type lives in `src/types/common.ts`, imported by every page) and render `<tr>` children directly; `Table` renders the `<thead>` from `columns`, wraps everything in a scroll container, and uses `table-layout: fixed`. **Every column except exactly one must have an explicit `width`** (px or `%`) — leaving two or more columns unconstrained makes column widths jump around whenever the row content/filter changes (this exact bug happened before and was fixed by auditing every `columns` array). The scroll container (`.scroll` in `Table.module.css`) is `flex: 1` inside a flex-column page body and scrolls **both** directions with a hidden scrollbar (`scrollbar-width: none` + `::-webkit-scrollbar{display:none}` — scrolling still works via wheel/touch/drag, just no visible track) and a `position: sticky` `<thead>` — this is what lets `AdminLayout`'s `.main` stay a **fixed** height (`--shell-content-height`, not just a floor) regardless of page size/row count, see **Consistent page chrome** below.

**Consistent page chrome via CSS vars**: `--shell-content-height` keeps every list page's white container a **fixed** height (`AdminLayout.module.css`'s `.main`) regardless of row count — the page's root div and `Table`'s scroll wrapper are both flex children (`.main > *` and `.scroll` respectively) so extra rows scroll *inside* `Table`, they don't grow `.main` taller. This matters beyond visual consistency: `.bodyInner` uses `align-items: stretch`, so `Sidebar` stretches to match `.main`'s height — if `.main` were allowed to grow (the old `min-height` behavior), the Sidebar/logout button would visibly shift position whenever a page had more rows or a larger page size selected (a real bug, fixed by making the height fixed + the table scroll internally instead of pushing the container taller). `--table-row-height` keeps table rows a fixed height; `--page-title-height` keeps a page's title row the same height whether or not it includes a "Yeni X" add button. When adding a new list page, reuse these vars rather than hand-tuning heights, or the pages will visually drift apart again.

**`shared/components/ActionMenu`**: a 3-dot trigger whose dropdown is rendered via `createPortal(..., document.body)` — required because table cells sit inside a scroll container that would otherwise clip an absolutely-positioned dropdown. Used only on **Campaigns / Categories / Products**; **Orders** and **Users** intentionally use a plain inline "Göstər" button + detail `Modal` instead (a deliberate product inconsistency, not an oversight — don't "fix" it by unifying them).

**CRUD modal pattern** (Campaigns, Categories, Products all repeat this shape — copy it for any new similar page): all the state behind it — `formOpen` / `editing` / `form` / `deleteTarget` / `viewTarget`, plus `openCreate`/`openEdit` — comes from `useCrudModal<TItem, TForm>(emptyForm, toForm)` (`src/shared/hooks/useCrudModal.ts`), not hand-rolled per page. `toForm(item: TItem): TForm` maps a list item to the form shape (mirrors the page's own `emptyForm` shape); `openCreate(overrides?: Partial<TForm>)` takes an optional partial-form object for cases like Products, where the create form needs a default `category_id` computed from data the hook itself doesn't know about (`openCreate({ category_id: categoryOptions[0]?.id ?? '' })`). On top of that:
- one `Modal` for create/edit, driven by `formOpen` / `editing` / `form`,
- one `ConfirmModal` for delete, driven by `deleteTarget`,
- one `wide` `Modal` for a read-only detail view, driven by `viewTarget`, opened from `ActionMenu`'s "Bax".

Pagination (`page`/`pageSize`/`paged`) on every list page — including Orders and Users, which don't use `useCrudModal` — comes from `usePagination<T>(filteredItems, initialPageSize?)` (`src/shared/hooks/usePagination.ts`, default `initialPageSize` is 5 — deliberately kept at 5, a larger default was tried and reverted because it visually crowded the fixed-height page chrome described above): slices `filteredItems` client-side and resets `page` to `1` whenever `setPageSize` is called (Orders is the only page with a page-size selector, via `Pagination`'s `onPageSizeChange`).

The "Şəkil ünvanı" field in these forms is a plain text input bound to `form.imageUrl`, mapped to/from the API's `img_url` by the relevant `src/lib/adapters/*.ts` file. Thumbnails are rendered by the shared `Thumbnail` component (`imageUrl`/`image`/`color` props): `<img>` if `imageUrl`/`img_url` is set, otherwise a fixed decorative emoji+color pair defined per-resource in the adapter (the API itself carries no emoji/color field — there are no real image files behind most items yet). `src/services/uploadService.ts` (`POST /upload`, multipart) exists for future use but is not wired into any form UI yet — the image field stays a manual URL text input for now.

**Search flow**: the raw search string lives in `AdminLayout` (`useState`, passed to `Header`'s controlled input so typing feels instant), but the value handed down to pages via `<Outlet context={{ search } satisfies LayoutOutletContext}>` is `useDebounce(search, 500)` — so every page's `useMemo` filter (reading `search` via `useOutletContext<LayoutOutletContext>()`) only re-runs 500ms after the user stops typing. Don't debounce inside individual pages; the debounce point is centralized in `AdminLayout` so all pages get it for free. `LayoutOutletContext` (`{ search: string }`) lives in `src/types/common.ts` — it's the one type shared between the layout (producer) and every protected page (consumer), see **TypeScript**.

**`useTitle(title)`** (`src/shared/hooks/useTitle.ts`): sets `document.title` to `"{title} · Tik Tak Admin"` for the life of the component and restores the previous title on unmount. Every page component calls it once, near the top, with its own Azerbaijani title (e.g. `useTitle('Sifarişlər')`) — do the same for any new page.

## TypeScript

The project migrated from plain JS/JSX to TypeScript in full (`tsconfig.json`, `strict: true` + `noUncheckedIndexedAccess` + `noImplicitOverride`, `allowJs: false` — there is nothing left for `allowJs` to cover). `npm run typecheck` (`tsc --noEmit`) is the only command that actually type-checks; `npm run build` transpiles via Vite/esbuild and will happily emit a broken bundle from code with type errors, so run `typecheck` (not just `build`) before calling a change done. `npm run lint` (oxlint) also covers `.ts`/`.tsx` natively — no `typescript-eslint`/ESLint in this repo — via the `typescript` plugin in `.oxlintrc.json`, which enforces `typescript/no-explicit-any: error` (see the `any`/`unknown` policy below) and warns on `typescript/consistent-type-imports` (the hard enforcement of that one is `verbatimModuleSyntax` in `tsconfig.json`, which makes a missing `import type` a compile error, not just a lint warning). There's a `src/vite-env.d.ts` for the Vite client types + the `ImportMetaEnv`/`VITE_API_BASE_URL` typing + one ambient `declare module '*.module.css'` (a single global CSS-Modules type declaration, not per-component `.d.ts` files — don't add another one).

**`src/types/` layout**: one file per resource (`category.ts`, `product.ts`, `campaign.ts`, `order.ts`, `user.ts`), plus `common.ts` (`BadgeColor`, `Column`, `IconComponent`, `LayoutOutletContext` — small types shared across *multiple* resources/components), `auth.ts` (`Profile`, `AuthTokens`, `LoginPayload`, `LoginResponse`), `api.ts` (the `declare module 'axios'` augmentation for `skipAuthRetry`/`_retry`, plus `ApiEnvelope<T>`/`UnwrappedApi`), and a type-only `index.ts` barrel (pages may import from the barrel; `lib/adapters/*.ts` and `services/*.ts` import directly from the specific resource file to keep the dependency graph flat). Each resource file follows the same four-type shape — copy this pattern for any new resource:
- `XApi` — the raw backend shape (snake_case fields like `img_url`/`created_at`, matching `docs/API.md` exactly).
- `X` — the UI/domain shape `mapXFromApi` produces (camelCase, e.g. `imageUrl`/`date`).
- `XForm` — the shape a create/edit `<Modal>` form holds in state (usually close to `X` but not identical — e.g. `ProductForm.category_id: number | string` is deliberately wider than `Product.category_id: number | ''`, because a `<select>`'s `onChange` always hands back a plain string while `openCreate`'s seed value is `number | ''`; this was a real type mismatch TypeScript caught during the migration, not a hypothetical one).
- `XPayload` — what `mapXToApi` sends on create/update (usually `Pick<XApi, ...>` of just the writable fields, no server-generated `id`/`created_at`).

**Enums are derived, not redefined**: `OrderStatus` and `ProductType` are NOT declared in `src/types/` — they're `keyof typeof ORDER_STATUS_LABELS` / `keyof typeof PRODUCT_TYPE_LABELS`, computed directly in `src/lib/constants/orderStatus.ts`/`productTypes.ts` (which `as const` their label objects to get literal-string keys) and re-exported type-only from `src/types/order.ts`/`product.ts`. If you add a new order status or product type, add it to the `_LABELS` object in `lib/constants/` — the type updates itself, nowhere else needs touching.

**The `any`/`unknown`/generics policy** (the rules actually enforced, not just a preference):
- **No bare `any` anywhere in hand-written code** — enforced by `typescript/no-explicit-any: error` in oxlint, not just a style guide entry.
- **`unknown` belongs at exactly three boundaries**: a `catch (err)` block (already `unknown` by default under `strict` — narrow with `err instanceof Error` before reading `.message`, e.g. `Login.tsx`'s submit handler; never re-annotate as `any`), `JSON.parse` results (`lib/auth/session.ts`'s `getStoredProfile()` does `JSON.parse(raw) as Profile` — a documented assertion, not a runtime validator, since only this module ever writes that localStorage key), and the axios response body inside `handleSuccess` before the envelope is unwrapped. Don't reach for `unknown` anywhere else as a stand-in for "didn't figure out the real type."
- **Generics exist on exactly five things**: `usePagination<T>`, `useCrudModal<TItem, TForm>`, `useDebounce<T>`, and the unwrapped `api.get<T>/post<T>/put<T>/delete<T>` calls in every `services/*.ts` file. `Table`/`Column` is deliberately **not** generic — `columns[].key` is only ever used as a React `key` + display label, never to index into row data (each page hand-writes its own `<tr>`), so a `Column<TRow>` would be speculative complexity with no bug it actually catches. Adapters are five separate concrete functions, not one generic `mapFromApi<TApi, TDomain>` — the per-resource field-renaming logic is bespoke enough that a generic mapper would type-check while doing nothing useful.
- **One sanctioned double-assertion, nowhere else**: `src/services/axiosInstance.ts` exports `api as unknown as UnwrappedApi` at the bottom of the file, because `handleSuccess` reshapes every response at runtime (unwraps the envelope) in a way axios's own `AxiosInstance` type can't express — the real `AxiosInstance` type and the post-interceptor shape are structurally incompatible by design, not by accident. `handleSuccess`'s own return type annotation (`AxiosResponse`, not what it actually returns) exists for the same reason — to satisfy axios's interceptor typing at the registration call, with a comment pointing at the real contract (`UnwrappedApi`). Don't copy this `as unknown as X` pattern elsewhere; if you hit a similar impedance mismatch, ask whether it's genuinely structural (like this one) before reaching for a cast.

**Other spots worth knowing before touching typed code**: `Button`'s `ButtonProps extends ComponentPropsWithRef<'button'>` (not `forwardRef`) — `ConfirmModal` passes `ref={cancelBtnRef}` to `Button`, and `ComponentPropsWithRef` is what lets a plain function component accept `ref` under React 19 without wrapping it. `ErrorBoundary`'s `override` keywords on `state`/`componentDidCatch`/`render` are required by `noImplicitOverride` since all three actually override members of React's `Component` base class. `noUncheckedIndexedAccess` is on — lookup tables like `ORDER_STATUS_LABELS[status]`/`PRODUCT_TYPE_LABELS[type]` are already used defensively (`?? ''`, or the key is provably non-optional going in), so this flag hasn't required new guards, just don't remove the existing ones assuming they're now redundant.
